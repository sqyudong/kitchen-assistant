## 1\. 角色与总任务

你是一位资深全栈工程师，精通原生 JavaScript 单页应用开发、Supabase（PostgREST）后端集成与大模型 Function-Calling 式交互设计。
请为我开发一个名为「食材管理」的移动端优先单页 Web 应用。最终交付物是【一个可直接双击运行的 index.html 文件】，内含全部 HTML、CSS、JavaScript，不依赖任何构建工具（无 npm、无 webpack、无框架）。
产品定位：面向小型餐饮店、食堂档口、家庭厨房等场景的通用食材库存管理工具。支持多用户、多仓库。核心卖点是"说人话就能管库存"——用户在输入框中打字输入"今天买了5斤土豆放冰箱"，AI 自动完成结构化入库，无需学习任何表单操作。

## 3\. 核心概念与权限体系

### 3.1 角色定义

|角色|说明|
|-|-|
|系统管理员（admin）|拥有最高权限，可查看和管理所有仓库、所有用户账号|
|普通用户（user）|可创建自己的仓库，可被邀请加入他人仓库|

### 3.2 仓库角色

|角色|说明|
|-|-|
|老板（owner）|仓库创建者，每个仓库有且仅有一个老板，拥有该仓库完整管理权（管理库存、管理成员、删除仓库）|
|员工（staff）|被老板邀请加入的用户，可管理库存，但不能管理成员，不能删除仓库|

|操作|系统管理员|仓库老板|仓库员工|

### 3.4 关键约束

* 一个用户可以创建多个仓库，也可以被邀请加入多个仓库；
* 每个仓库有且仅有一个老板，老板可指定员工成为新老板，则原老板变成员工；
* 老板通过输入员工用户名来邀请员工（员工必须已注册）；
* 系统管理员可管理一切，不受仓库归属限制。

### 4.4 数据流原则

云端为唯一数据源，本地不做任何业务数据缓存；
所有写操作流程：先写云端 → 成功后更新内存数据对象 → 最后重新渲染界面；
云端请求失败时 toast 提示"网络异常，请重试"，不做离线降级；
页面每次加载/刷新时，必须从云端重新拉取最新数据；
所有业务数据查询必须携带 warehouse\_id 过滤条件（admin 查看全部时除外）。

## 6\. 功能模块规格

### 6.1 注册模块

用户打开注册页，输入用户名、密码、确认密码；
前端校验：用户名非空且不含特殊字符、密码≥6位、两次密码一致；
查询云端 accounts 表检查用户名是否已存在：
已存在 → 提示"用户名已被占用"；
不存在 → 密码明文写入 accounts 表（role 默认为 user）；
注册成功 → toast 提示 → 自动跳转到登录页。

### 6.2 登录模块

用户输入用户名和密码，点击登录；
先查询 accounts 表中是否存在该用户名：
不存在 → 提示 "用户名错误"；
存在 → 将输入密码与存储值比对：
不匹配 → 提示 "密码错误"；
匹配 → 登录成功；
登录成功后：
写入 ks\_current\_user（含 id、username、role）；
若勾选"记住密码"→ 保存到 ks\_saved\_accounts；
若勾选"自动登录"→ 设置 ks\_auto\_login 为 "1"；
查询 warehouse\_members 获取该用户所属仓库列表，自动选中上次仓库（ks\_current\_warehouse）或第一个仓库；
若用户无任何仓库 → 引导创建第一个仓库；
记录登录日志到云端。

### 6.3 退出登录

顶部状态栏显示"退出"按钮；
点击后清除 ks\_current\_user 和 ks\_auto\_login；
跳转回登录页。

### 6.4 修改密码

用户在设置面板中点击"修改密码"；
弹出表单：旧密码 + 新密码 + 确认新密码；
校验旧密码是否正确：
错误 → 提示"旧密码错误"；
正确 → 校验新密码≥6位、两次一致；
更新 accounts 表中的 password 字段；
提示"密码修改成功"。

### 6.5 仓库管理

#### 创建仓库

用户点击顶部"+"按钮，弹出输入框填写仓库名称；
写入 warehouses 表（owner\_id = 当前用户 ID）；
同时写入 warehouse\_members 表（role = owner）；
自动初始化 4 个默认位置（常温货架、冰箱冷藏、冰箱冷冻、调料柜）写入 locations 表；
创建成功 → 自动切换到该仓库 → 重新拉取数据 → 渲染。

#### 切换仓库

用户点击顶部仓库名称 → 弹出仓库列表；
列表来源：
普通用户：warehouse\_members 中 user\_id = 当前用户的所有仓库；
admin：warehouses 表全部仓库；
每个仓库项显示：仓库名 + 角色标签（老板/员工）；
点击某仓库 → 更新 ks\_current\_warehouse → 重新拉取该仓库数据 → 渲染。

#### 删除仓库

仅老板或 admin 可操作；
二次确认弹窗："删除后该仓库所有数据不可恢复，确认删除？"；
确认后级联删除：inventory、locations、recipes、logs、warehouse\_members、warehouses 中该仓库的所有记录。

### 6.6 仓库成员管理

#### 添加员工（仅老板或 admin）

在设置面板中点击"添加员工"；
输入员工用户名 → 查询 accounts 表：
用户名不存在 → 提示"该用户名不存在，请让对方先注册"；
存在 → 检查 warehouse\_members 是否已有记录：
已是成员 → 提示"该用户已在仓库中"；
不是 → 写入 warehouse\_members（role = staff）→ 提示"添加成功"。

#### 移除员工（仅老板或 admin）

在成员列表中点击"移除"按钮；
删除 warehouse\_members 中对应记录；
提示"已移除"。

#### 查看成员列表

显示该仓库所有成员（老板 + 员工），标注角色；
员工视角下不显示"添加"和"移除"按钮。

### 6.7 系统管理员专属功能

双击顶部角色标签 → 打开管理面板（仅 admin 可见）；
查看所有用户列表，可删除用户（同时清理该用户在 warehouse\_members 中的记录）；
Excel 批量导入用户：SheetJS 解析上传文件，按列（用户名/密码）批量写入 accounts 表，导入结果汇总 toast；
查看所有仓库列表；
一键清除本机缓存的全部登录凭证。

### 6.8 库存管理

渲染库存列表：按入库时间倒序；受位置筛选条件影响；默认折叠显示前 N 条，"展开全部"状态持久化；
手动新增食材：表单弹窗（名称/数量/位置）；提交时查重，重名自动追加序号（土豆 → 土豆2 → 土豆3）；
删除食材：删除并推入撤回队列（附删除时间戳）；
撤回删除：5 分钟内可撤回，重新写入云端；
快速改位置：点选式弹出位置列表，点选即改；
过期预警规则：
常温位置（is\_cold = false）：入库超 24 小时为黄档，超 48 小时为红档；
低温位置（is\_cold = true）：入库超 7 天为黄档；
入库时长人性化显示（"3天"、"5小时"、"20分钟"）。

### 6.9 AI 文本录入（核心模块）

用户在底部输入框打字，点击发送按钮提交文本；
提交后进入 AI 分析流程：
清洗用户文本（去语气词"嗯""啊""那个"、标点归一）；
拼装 System Prompt（见第 7 节），注入当前仓库的库存清单与位置列表作为上下文；
请求 DeepSeek，强制 JSON 输出，超时 15s，失败提示"AI 未能理解，请手动操作"；
解析返回 JSON。
按返回的 action 字段分发执行：
add\_stock → 入库（多条则循环），toast 确认；
use\_stock → 扣减/删除对应库存；
move\_stock → 改位置；若只移走部分数量，原记录加警告标记；
new\_location → 弹窗询问用户是否新增该位置到当前仓库；
ask\_qty → 在输入框上方以气泡形式反问用户数量，等待补充输入；
chat → 普通对话回复，直接气泡展示。
每次成功执行均写入操作日志（归属当前仓库）；
AI 处理期间，状态栏圆点变为橙色，输入框禁用，防止重复提交。

### 6.10 菜谱管理

新增菜谱：弹窗表单（名称、食材清单、步骤），归属当前仓库；
删除菜谱；
菜谱列表渲染：卡片式展示，点击可展开查看步骤。

### 6.11 日志模块

写日志：同时写云端（含 warehouse\_id）+ 追加本地日志缓存；
渲染日志：时间倒序；删除类日志在 5 分钟内显示「撤回」按钮（倒计时结束自动移除按钮）。

### 7\. AI System Prompt 规格（嵌入分析流程中使用）

你是"食材管理"系统的智能库存助手。用户会用口语化的中文描述进货、用料、移库。
请结合【当前仓库】【当前库存】与【已有位置】，输出且仅输出一个 JSON 对象，字段如下：
{
"action": "add\_stock | use\_stock | move\_stock | new\_location | ask\_qty | chat",
"items": \[{"item": "食材名", "qty": "数量+单位", "loc": "位置名"}],
"reply": "给用户的自然语言回复（简短、口语化）"
}

规则：

1. 用户说"买了/进了/到货/来了"→ add\_stock；
说"用了/做了/吃掉/消耗"→ use\_stock；
说"移到/放到/挪到/换到"→ move\_stock。
2. 位置必须从【已有位置】中选择最接近的；若无匹配，action 设为 new\_location
并在 items\[0].loc 给出建议位置名。
3. 缺少数量时 action 设为 ask\_qty，并在 reply 中礼貌追问。
4. 一句话含多个食材时，items 输出多条。
5. 与库存无关的闲聊，action 设为 chat，仅填 reply。
6. reply 字段必须填写，语气简洁友好。

【当前仓库】：{动态注入当前仓库名称}
【当前库存】：{动态注入当前仓库的 inventory 列表}
【已有位置】：{动态注入当前仓库的 locations 列表}

### 8\. 边界条件与异常处理（必须实现）

DeepSeek 返回非法 JSON → 重试 1 次，仍失败则提示"AI 未能理解，请手动操作"。
Supabase 请求失败 → toast 提示"网络异常，请重试"，不做离线降级。
撤回队列每次渲染时清理过期（>5min）条目。
所有用户输入与 AI 返回内容插入 DOM 前必须转义，防 XSS。
网络请求统一超时设置（Supabase 10s，DeepSeek 15s）。
并发写入保护：同一食材的增删操作需串行执行，避免竞态。
AI 处理期间禁用输入框与发送按钮，防止重复提交。
页面加载时若云端不可达，显示明确的错误提示页（非空白页），并提供"重试"按钮。
用户被移除出仓库后，若当前正在该仓库 → 下次刷新时自动切换到其他可用仓库或引导创建新仓库。
删除用户时级联清理 warehouse\_members 中该用户的所有记录。
删除仓库时级联清理所有关联表数据。
注册时用户名长度限制 2-20 字符，仅允许中文、英文、数字、下划线。
